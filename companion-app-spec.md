## 1. Project Context: What is SpendWiser?

**SpendWiser** is a personal finance tracking Progressive Web App (PWA) built with:

- **Frontend:** React 18, TypeScript, Vite, Tailwind CSS
- **Backend:** Firebase (Authentication via Google OAuth, Firestore for data storage)
- **Hosting:** Netlify (static hosting + serverless functions)
- **Live URL:** https://spenditwiser.netlify.app/

### Core Features of SpendWiser

SpendWiser allows users to:

- **Track transactions** (income & expenses) with categories, tags, and comments
- **Manage multiple bank accounts** and credit cards with real-time balance tracking
- **Set budgets** per category and track spending against limits
- **Financial goals** with progress tracking
- **Loan management** with EMI tracking
- **Recurring transactions** that auto-generate on schedule
- **Transaction shortcuts** for quick data entry (e.g., type `coffee.150` to auto-fill)
- **Monthly reports** with PDF export
- **CSV import/export** for data portability
- **Push notifications** for daily transaction reminders
- **Dark mode**, custom fonts, timezone support
- **PWA** installable on mobile/desktop with offline capabilities

### Firestore Data Model (High Level)

All user data lives under: `spenders/{userId}/`

```
spenders/
  └── {userId}/                    ← Firebase Auth UID
      ├── transactions/            ← Income & expense records
      ├── accounts/                ← Bank accounts & credit cards
      ├── budgets/                 ← Category budget limits
      ├── goals/                   ← Savings goals
      ├── loans/                   ← Active loans with EMI
      ├── recurring_transactions/  ← Auto-generating transactions
      ├── shortcuts/               ← Quick-entry shortcuts
      ├── pending_transactions/    ← ⭐ NEW: SMS-parsed drafts (Phase 1)
      └── (user doc fields)        ← Settings, preferences, API key
```

### User Document Fields (Relevant)

The user document at `spenders/{userId}` contains:

```typescript
{
  uid: string;
  email: string;
  displayName: string;
  currency: string;              // e.g., "₹", "$"
  themePreference: 'dark' | 'light';
  listenerApiKey?: string;       // ⭐ The API key for the companion app
  // ... other settings
}
```

The `listenerApiKey` is the field the companion app uses to authenticate with the webhook. It's a UUID v4 string generated by the user in the SpendWiser Settings page.

## 2. The SMS Automation Feature: Big Picture

### The Problem

Users manually type every transaction into SpendWiser. This is tedious, especially for frequent small purchases. Every time the bank sends an SMS alert like:

> "Your ac XX1234 is debited for Rs.450.00 on 05-Apr-26 at Swiggy Avl bal Rs.12340.50"

...the user has to open SpendWiser, click "Add Transaction", type "Swiggy", type "450", select "Expense", pick a category, and save. This friction leads to missed transactions.

### The Solution

Build a **companion Android app** that:

1. Silently listens for incoming SMS messages in the background
2. Filters for bank transaction alerts using keyword matching
3. Forwards the raw SMS text to a Netlify serverless webhook
4. The webhook parses the SMS (using Google Gemini AI + regex fallback), extracts amount/merchant/type
5. Saves a "pending transaction" to Firestore
6. The SpendWiser web dashboard instantly shows a notification banner via real-time Firestore listener
7. The user reviews, edits if needed, and approves with one tap

**Result:** Zero manual data entry. The transaction appears on the dashboard within seconds of receiving the bank SMS.

### Why a Separate App?

- **Web browsers cannot read SMS** — this is a hard OS-level restriction
- **Google Play Store restricts SMS permissions** — only apps whose core functionality requires SMS (like messaging apps) are approved. A finance tracker would be rejected.
- **Sideloading bypasses this** — since SpendWiser is a personal/small-user-base project, users can install the APK directly (sideload). They trust the code because it's open source.

## 4. What's Already Built (Phase 1 — The Web Backend)

Phase 1 is **fully implemented and tested** in the SpendWiser web app repository. Here's exactly what was built:

### 4.1 Data Schema (`src/types/types.ts`)

A new `PendingTransaction` interface was added:

```typescript
export interface PendingTransaction {
  id: string;
  smsText: string;
  extractedAmount?: number;
  extractedMerchant?: string;
  extractedType?: 'income' | 'expense';
  status: 'pending' | 'approved' | 'rejected';
  receivedAt: string; // ISO date string
}
```

The `User` type was extended with:

```typescript
export type User = {
  // ... existing fields
  listenerApiKey?: string; // API Key for the SMS companion app
};
```

### 4.2 API Key Management UI (`AutomationSettings.tsx`)

A new "SMS Automation" settings section was added to the Settings page. It allows users to:

- **Generate** a new API key (UUID v4 via `crypto.randomUUID()`)
- **Copy** the key to clipboard (for pasting into the companion app)
- **Regenerate** the key (invalidates the old one)
- **Revoke** the key (with confirmation dialog — stops the companion app from working)

The key is stored in the user's Firestore document as `listenerApiKey`.

### 4.3 The Webhook (`netlify/functions/process-sms.ts`)

A Netlify serverless function that:

1. Accepts only `POST` requests
2. Parses JSON body: `{ apiKey: string, smsText: string }`
3. Validates the API key by querying Firestore: `WHERE listenerApiKey == apiKey`
4. Parses the SMS text using **Google Gemini 2.5 Flash** (with regex fallback)
5. Saves the result to `spenders/{userId}/pending_transactions`

**Full source code of the webhook:**

```typescript
import { Handler } from '@netlify/functions';
import * as admin from 'firebase-admin';
import { GoogleGenerativeAI } from "@google/generative-ai";

// Initialize Firebase Admin if not already initialized
if (!admin.apps.length) {
  try {
    admin.initializeApp({
      credential: admin.credential.cert({
        projectId: process.env.VITE_FIREBASE_PROJECT_ID || process.env.FIREBASE_PROJECT_ID,
        clientEmail: process.env.FIREBASE_CLIENT_EMAIL,
        privateKey: process.env.FIREBASE_PRIVATE_KEY?.replace(/\\n/g, '\n'),
      }),
    });
  } catch (error) {
    console.error('Firebase Admin initialization error:', error);
  }
}

const db = admin.firestore();

export const handler: Handler = async (event) => {
  if (event.httpMethod !== 'POST') {
    return { statusCode: 405, body: 'Method Not Allowed' };
  }

  try {
    const body = JSON.parse(event.body || '{}');
    const { apiKey, smsText } = body;

    if (!apiKey || !smsText) {
      return { statusCode: 400, body: JSON.stringify({ error: 'Missing apiKey or smsText' }) };
    }

    // 1. Validate API Key
    const usersRef = db.collection('spenders');
    const snapshot = await usersRef.where('listenerApiKey', '==', apiKey).limit(1).get();

    if (snapshot.empty) {
      return { statusCode: 401, body: JSON.stringify({ error: 'Invalid API Key' }) };
    }

    const userDoc = snapshot.docs[0];
    const userId = userDoc.id;

    // 2. Parse SMS Text using Gemini API
    let amount: number | undefined;
    let merchant = 'Unknown Merchant';
    let type: 'expense' | 'income' = 'expense';

    try {
      const geminiApiKey = process.env.VITE_GEMINI_API_KEY || process.env.GEMINI_API_KEY;
      if (!geminiApiKey) {
        throw new Error('Missing Gemini API Key');
      }
      
      const genAI = new GoogleGenerativeAI(geminiApiKey);
      const model = genAI.getGenerativeModel({ model: 'gemini-2.5-flash' });

      const prompt = `Extract from this bank SMS — amount (number), merchant (string), type ("income" or "expense"). Reply in JSON only, no markdown. SMS: "${smsText}"`;

      const result = await model.generateContent(prompt);
      const responseText = result.response.text().trim();
      
      // Strip markdown code block if model mistakenly added it
      const jsonString = responseText.replace(/```json\n?|\n?```/g, '');
      const parsedData = JSON.parse(jsonString);

      amount = typeof parsedData.amount === 'number' ? parsedData.amount : parseFloat(parsedData.amount);
      merchant = parsedData.merchant || 'Unknown Merchant';
      type = parsedData.type === 'income' ? 'income' : 'expense';
      
    } catch (parseError) {
      console.error('Error parsing SMS with Gemini, falling back to Regex:', parseError);
      
      // Fallback Regex
      const amountMatch = smsText.match(/(?:rs\.?|inr|\$)\s*([\d,]+\.?\d*)/i) ||
                          smsText.match(/([\d,]+\.?\d*)\s*(?:rs\.?|inr|\$)/i);
      amount = amountMatch ? parseFloat(amountMatch[1].replace(/,/g, '')) : undefined;
      
      type = smsText.toLowerCase().match(/(debited|spent|paid)/) ? 'expense' : 
             smsText.toLowerCase().match(/(credited|received|refunded)/) ? 'income' : 'expense';

      const merchantMatch = smsText.match(/(?:at|to|on)\s+([A-Za-z0-9\s]+?)(?=\s|$|\.)/i);
      merchant = merchantMatch ? merchantMatch[1].trim() : 'Unknown Merchant';
    }

    // 3. Save to Firestore pending_transactions
    const pendingRef = db.collection(`spenders/${userId}/pending_transactions`);
    await pendingRef.add({
      smsText,
      extractedAmount: amount,
      extractedMerchant: merchant,
      extractedType: type,
      status: 'pending',
      receivedAt: new Date().toISOString()
    });

    return {
      statusCode: 200,
      body: JSON.stringify({ success: true, message: 'Transaction queued for review.' }),
    };

  } catch (error: any) {
    console.error('Error processing SMS:', error);
    return {
      statusCode: 500,
      body: JSON.stringify({ error: 'Internal Server Error', details: error.message }),
    };
  }
};
```

### 4.4 Dashboard Integration (`App.tsx`)

**Real-time listener** for pending transactions:

```typescript
useEffect(() => {
  if (user) {
    const pendingRef = collection(db, 'spenders', user.uid, 'pending_transactions');
    const unsubscribe = onSnapshot(pendingRef, snapshot => {
      const pendingData = snapshot.docs.map(doc => ({
        id: doc.id, ...doc.data()
      })) as PendingTransaction[];
      setPendingTransactions(pendingData);
    });
    return () => unsubscribe();
  }
}, [user]);
```

**Review handler** that pre-fills the AddTransactionModal:

```typescript
const handleReviewPending = (pending: PendingTransaction) => {
  setActivePendingTransactionId(pending.id);
  setPreSelectedTransactionType(pending.extractedType || 'expense');
  setPreFilledTransactionData({
    name: pending.extractedMerchant || '',
    amount: pending.extractedAmount || 0,
    date: pending.receivedAt,
    type: pending.extractedType || 'expense',
  });
  setIsAddTransactionModalOpen(true);
};
```

**Cleanup** after the user approves a pending transaction:

```typescript
// Inside handleAddTransaction, after saving:
if (activePendingTransactionId) {
  const pendingRef = doc(db, 'spenders', user.uid, 'pending_transactions', activePendingTransactionId);
  await deleteDoc(pendingRef);
  setActivePendingTransactionId(null);
  setPreFilledTransactionData(undefined);
}
```

**Persistent banner/alert** rendered above all page content:

```tsx
{pendingTransactions.length > 0 && currentScreen !== 'settings' && (
```

## 5. The Webhook API Contract

This is the **exact API** the companion app must call.

### Endpoint

```
POST https://spenditwiser.netlify.app/.netlify/functions/process-sms
```

For local testing (with `netlify dev`):
```
POST http://localhost:8888/.netlify/functions/process-sms
```

### Request

```http
POST /.netlify/functions/process-sms HTTP/1.1
Content-Type: application/json

{
  "apiKey": "307c4de1-f94b-46ad-9eaf-882f383d807c",
  "smsText": "Your ac XX1234 is debited for Rs.9990.00 on 05-Apr-26 at Hariharen Avl bal Rs.12340.50"
}
```

| Field     | Type   | Required | Description                                        |
|-----------|--------|----------|----------------------------------------------------|
| `apiKey`  | string | ✅ Yes   | UUID v4 generated from SpendWiser Settings page    |
| `smsText` | string | ✅ Yes   | The raw, unmodified SMS body text from the phone   |

### Responses

**200 OK — Success:**
```json
{
  "success": true,
  "message": "Transaction queued for review."
}
```

**400 Bad Request — Missing fields:**
```json
{
  "error": "Missing apiKey or smsText"
}
```

**401 Unauthorized — Invalid API key:**
```json
{
  "error": "Invalid API Key"
}
```

**405 Method Not Allowed — Not a POST:**
```
Method Not Allowed
```

**500 Internal Server Error — Server-side failure:**
```json
{
  "error": "Internal Server Error",
  "details": "error message string"
}
```

## 6. Firestore Data Schema Reference

```typescript
{
  uid: "firebase-auth-uid-string",
  email: "user@example.com",
  displayName: "John Doe",
  currency: "₹",
  listenerApiKey: "307c4de1-f94b-46ad-9eaf-882f383d807c",
  // ... other fields not relevant to companion app
}
```

### `spenders/{userId}/pending_transactions/{docId}`

This is what the webhook creates. The companion app does NOT write to Firestore directly — it only calls the webhook.

```typescript
{
  smsText: "Your ac XX1234 is debited for Rs.450.00 on 05-Apr-26 at Swiggy Avl bal Rs.12340.50",
  extractedAmount: 450,
  extractedMerchant: "Swiggy",
  extractedType: "expense",     // "income" or "expense"
  status: "pending",            // Always "pending" when created
  receivedAt: "2026-04-05T13:12:00.000Z"  // ISO 8601 UTC
}
```

### `spenders/{userId}/transactions/{docId}`

This is the final transaction after user approval (the companion app never touches this):

```typescript
{
  name: "Swiggy",
  amount: -450,                 // Negative for expenses, positive for income
  date: "2026-04-05",          // YYYY-MM-DD format
  category: "Food & Dining",
  type: "expense",
  accountId: "account-doc-id",
  createdAt: "2026-04-05T13:15:00.000Z",
  // optional: comments, tags, loanId, etc.
}
```

## 7. Phase 2: Android Companion App — Full Spec

### App Name
**SpendWiser Listener** (or "SpendWiser SMS")

### App Description
A minimalist Android companion app that silently listens for bank SMS alerts and forwards them to your SpendWiser dashboard. Requires sideloading (not on Play Store).

### Core Requirement
The app must be able to **receive SMS messages in the background** (even when the app is closed or the phone is locked) and **send an HTTP POST** to the webhook.

### Target Android Version
- **Minimum SDK:** Android 8.0 (API 26) — for foreground service support
- **Target SDK:** Android 14 (API 34) — latest stable

## 8. Technology Stack & Tooling

### Recommended: React Native with Expo (Development Build)

Since the SpendWiser web app is written in React/TypeScript, using React Native allows code reuse and familiar tooling.

- **Framework:** React Native 0.76+ with Expo SDK 52+
- **Build Type:** Expo Development Build (NOT Expo Go — we need native modules for SMS)
- **Language:** TypeScript
- **Styling:** React Native StyleSheet (no web CSS)
- **Persistent Storage:** `@react-native-async-storage/async-storage`
- **HTTP Client:** Built-in `fetch` API
- **SMS Access:** `react-native-get-sms-android` or custom native module
- **Background Processing:** `react-native-background-actions` or Headless JS
- **Build:** EAS Build (`eas build --platform android --profile production`)

### Alternative: Kotlin/Jetpack Compose (Native Android)

If you prefer a fully native approach:

- **Language:** Kotlin
- **UI:** Jetpack Compose (Material 3)
- **SMS:** Android's native `BroadcastReceiver` for `SMS_RECEIVED`
- **HTTP:** OkHttp or Ktor
- **Storage:** SharedPreferences or DataStore
- **Background:** Foreground Service + Boot Receiver
- **Build:** Gradle → APK

> **Note:** The Kotlin approach gives more reliable background SMS processing and smaller APK size (~5MB vs ~30MB for React Native). Since this app is extremely simple (1 screen, no complex UI), native Kotlin may be the pragmatic choice.

## 12. Feature 3: SMS Broadcast Receiver

### How Android SMS Reception Works

When an SMS arrives, Android broadcasts an intent with action `android.provider.Telephony.SMS_RECEIVED`. By registering a `BroadcastReceiver`, the app can intercept and read the SMS.

### Registration in AndroidManifest.xml

```xml
<receiver
    android:name=".SmsReceiver"
    android:enabled="true"
    android:exported="true"
    android:permission="android.permission.BROADCAST_SMS">
    <intent-filter android:priority="999">
        <action android:name="android.provider.Telephony.SMS_RECEIVED" />
    </intent-filter>
</receiver>
```

### BroadcastReceiver Implementation (Kotlin)

```kotlin
class SmsReceiver : BroadcastReceiver() {
    override fun onReceive(context: Context, intent: Intent) {
        if (intent.action != Telephony.Sms.Intents.SMS_RECEIVED_ACTION) return

        val messages = Telephony.Sms.Intents.getMessagesFromIntent(intent)
        val fullMessage = messages.joinToString("") { it.messageBody }
        val sender = messages.firstOrNull()?.originatingAddress ?: "Unknown"

        Log.d("SmsReceiver", "SMS from $sender: $fullMessage")

        // Pass to the filter & forwarder
        SmsProcessor.process(context, sender, fullMessage)
    }
}
```

### React Native Equivalent

With `react-native-get-sms-android` or a custom native module:

```typescript
import { NativeEventEmitter, NativeModules } from 'react-native';

const SmsListener = NativeModules.SmsListener;
const smsEmitter = new NativeEventEmitter(SmsListener);

smsEmitter.addListener('onSmsReceived', (event: { body: string; sender: string }) => {
  console.log('SMS received:', event.body);
  processSms(event.sender, event.body);
});
```

## 13. Feature 4: Local SMS Filtering

### Why Filter Locally?

Not every SMS is a bank transaction. The app should ONLY forward SMS messages that look like bank alerts. This:
- Avoids unnecessary webhook calls (saves Netlify free-tier invocations)
- Preserves user privacy (personal messages are never sent anywhere)
- Reduces latency (no network call for irrelevant SMS)

### Filter Strategy

Use a two-step filter:

#### Step 1: Sender Number Filter

Indian banks typically send SMS from short codes or specific sender IDs:

```typescript
const BANK_SENDER_PATTERNS = [
  /^[A-Z]{2}-[A-Z]{6}$/,     // e.g., "VM-HDFCBK", "AX-ICICIB"
  /^[A-Z]{2}-[A-Z]{5,}$/,     // Various bank codes
  /^[A-Z]{6,}$/,               // Some phones strip the prefix
  /^\d{6}$/,                   // 6-digit short codes
];

function isBankSender(sender: string): boolean {
  return BANK_SENDER_PATTERNS.some(pattern => pattern.test(sender));
}
```

#### Step 2: Message Content Filter

Check for transaction-related keywords:

```typescript
const TRANSACTION_KEYWORDS = [
  // Debit indicators
  'debited', 'debit', 'spent', 'paid', 'purchase',
  'withdrawn', 'withdrawal', 'transfer', 'sent',
  
  // Credit indicators
  'credited', 'credit', 'received', 'refund', 'refunded',
  'deposited', 'deposit', 'cashback',
  
  // Amount patterns
  'rs.', 'rs ', 'inr', 'rupee',
  
  // Account indicators
  'a/c', 'ac ', 'acct', 'account',
  'card', 'xxxx', 'xx',
];

const EXCLUDE_KEYWORDS = [
  'otp', 'one time password', 'verification code',
  'login', 'password', 'reset',
];

function isTransactionSms(message: string): boolean {
  const lower = message.toLowerCase();
  
  // Exclude OTPs and auth messages
  if (EXCLUDE_KEYWORDS.some(kw => lower.includes(kw))) {
    return false;
  }
  
  // Must contain at least one transaction keyword
  const hasKeyword = TRANSACTION_KEYWORDS.some(kw => lower.includes(kw));
  
  // Must contain an amount pattern (Rs., INR, or number with currency)
  const hasAmount = /(?:rs\.?|inr|₹)\s*[\d,]+/i.test(message) || 
                    /[\d,]+\.?\d*\s*(?:rs\.?|inr|₹)/i.test(message);
  
  return hasKeyword && hasAmount;
}
```

#### Combined Filter

```typescript
function shouldForwardSms(sender: string, message: string): boolean {
  return isBankSender(sender) || isTransactionSms(message);
}
```

> **Note:** We use OR logic — if EITHER the sender looks like a bank OR the message contains transaction keywords with amounts, we forward it. This catches edge cases where bank sender IDs vary by carrier.

## 14. Feature 5: Webhook Forwarding

### HTTP POST Implementation

When an SMS passes the filter, send it to the webhook:

```typescript
const WEBHOOK_URL = 'https://spenditwiser.netlify.app/.netlify/functions/process-sms';

interface ForwardResult {
  success: boolean;
  message?: string;
  error?: string;
}

async function forwardToWebhook(apiKey: string, smsText: string): Promise<ForwardResult> {
  try {
    const response = await fetch(WEBHOOK_URL, {
      method: 'POST',
      headers: {
        'Content-Type': 'application/json',
      },
      body: JSON.stringify({
        apiKey,
        smsText,
      }),
    });

    const data = await response.json();

    if (response.ok) {
      return { success: true, message: data.message };
    } else {
      return { success: false, error: data.error || `HTTP ${response.status}` };
    }
  } catch (error) {
    // Network error — need to queue for retry
    return { success: false, error: `Network error: ${error.message}` };
  }
}
```

### Timeout Handling

Set a reasonable timeout (10 seconds) to avoid blocking:

```typescript
const controller = new AbortController();
const timeoutId = setTimeout(() => controller.abort(), 10000);

try {
  const response = await fetch(WEBHOOK_URL, {
    method: 'POST',
    headers: { 'Content-Type': 'application/json' },
    body: JSON.stringify({ apiKey, smsText }),
    signal: controller.signal,
  });
  clearTimeout(timeoutId);
  // ... handle response
} catch (error) {
  clearTimeout(timeoutId);
  if (error.name === 'AbortError') {
    return { success: false, error: 'Request timed out' };
  }
  return { success: false, error: error.message };
}
```

## 15. Feature 6: Offline Queue with Retry

### Why?

If the phone has no internet when a bank SMS arrives, the app must queue the request and retry when connectivity returns.

### Queue Data Structure

```typescript
interface QueuedSms {
  id: string;           // UUID for this queue entry
  smsText: string;
  timestamp: string;    // When the SMS was received
  retryCount: number;
  lastRetryAt?: string;
  error?: string;
}
```

### Storage

Use AsyncStorage (React Native) or SharedPreferences (Kotlin) to persist the queue:

```typescript
import AsyncStorage from '@react-native-async-storage/async-storage';

const QUEUE_KEY = 'sms_queue';

async function addToQueue(smsText: string): Promise<void> {
  const queue = await getQueue();
  queue.push({
    id: generateUUID(),
    smsText,
    timestamp: new Date().toISOString(),
    retryCount: 0,
  });
  await AsyncStorage.setItem(QUEUE_KEY, JSON.stringify(queue));
}

async function getQueue(): Promise<QueuedSms[]> {
  const data = await AsyncStorage.getItem(QUEUE_KEY);
  return data ? JSON.parse(data) : [];
}

async function removeFromQueue(id: string): Promise<void> {
  const queue = await getQueue();
  const filtered = queue.filter(item => item.id !== id);
  await AsyncStorage.setItem(QUEUE_KEY, JSON.stringify(filtered));
}
```

### Retry Logic

```typescript
async function processQueue(apiKey: string): Promise<void> {
  const queue = await getQueue();
  if (queue.length === 0) return;

  for (const item of queue) {
    // Skip if max retries reached (5 attempts)
    if (item.retryCount >= 5) continue;
    
    const result = await forwardToWebhook(apiKey, item.smsText);
    
    if (result.success) {
      await removeFromQueue(item.id);
      logActivity('success', item.smsText);
    } else {
      // Update retry count
      item.retryCount += 1;
      item.lastRetryAt = new Date().toISOString();
      item.error = result.error;
      await updateQueueItem(item);
    }
  }
}
```

### Connectivity-Based Retry

Listen for network state changes and trigger queue processing:

```typescript
// React Native
import NetInfo from '@react-native-community/netinfo';

NetInfo.addEventListener(state => {
  if (state.isConnected && state.isInternetReachable) {
    processQueue(apiKey);
  }
});
```

```kotlin
// Kotlin
class ConnectivityReceiver : BroadcastReceiver() {
    override fun onReceive(context: Context, intent: Intent) {
        val cm = context.getSystemService(Context.CONNECTIVITY_SERVICE) as ConnectivityManager
        val network = cm.activeNetwork
        if (network != null) {
            SmsQueueProcessor.processQueue(context)
        }
    }
}
```

## 16. Feature 7: Background Service & Persistence

### The Challenge

Android aggressively kills background apps to save battery. The SMS listener must survive:
- App being swiped away from recent apps
- Phone being locked for extended periods
- Phone restarting

### Solution: Foreground Service

A foreground service shows a persistent notification and tells Android "this app is doing important work — don't kill it."

#### Kotlin Implementation

```kotlin
class SmsListenerService : Service() {
    override fun onCreate() {
        super.onCreate()
        createNotificationChannel()
        startForeground(NOTIFICATION_ID, createNotification())
    }

    override fun onStartCommand(intent: Intent?, flags: Int, startId: Int): Int {
        return START_STICKY  // Restart if killed
    }

    private fun createNotification(): Notification {
        return NotificationCompat.Builder(this, CHANNEL_ID)
            .setContentTitle("SpendWiser Listener")
            .setContentText("Listening for bank SMS alerts...")
            .setSmallIcon(R.drawable.ic_notification)
            .setOngoing(true)
            .setPriority(NotificationCompat.PRIORITY_LOW)
            .build()
    }

    private fun createNotificationChannel() {
        val channel = NotificationChannel(
            CHANNEL_ID,
            "SMS Listener",
            NotificationManager.IMPORTANCE_LOW
        ).apply {
            description = "Keeps the SMS listener running in the background"
        }
        val manager = getSystemService(NotificationManager::class.java)
        manager.createNotificationChannel(channel)
    }

    override fun onBind(intent: Intent?): IBinder? = null

    companion object {
        const val CHANNEL_ID = "sms_listener_channel"
        const val NOTIFICATION_ID = 1
    }
}
```

### Auto-Start on Boot

Register a boot receiver to start the service when the phone restarts:

```xml
<receiver
    android:name=".BootReceiver"
    android:enabled="true"
    android:exported="true">
    <intent-filter>
        <action android:name="android.intent.action.BOOT_COMPLETED" />
    </intent-filter>
</receiver>
```

```kotlin
class BootReceiver : BroadcastReceiver() {
    override fun onReceive(context: Context, intent: Intent) {
        if (intent.action == Intent.ACTION_BOOT_COMPLETED) {
            val apiKey = PreferenceManager.getApiKey(context)
            if (apiKey != null) {
                val serviceIntent = Intent(context, SmsListenerService::class.java)
                ContextCompat.startForegroundService(context, serviceIntent)
            }
        }
    }
}
```

### Battery Optimization

Request the user to disable battery optimization for the app:


## 22. Security Considerations

### What Data Leaves the Phone?

Only the **raw SMS text** and the **API key** are sent to the webhook. The companion app:
- Does NOT send the phone number or IMEI
- Does NOT send the sender's number
- Does NOT store any SMS on any server (only in local activity log)
- Does NOT access any messages beyond the real-time broadcast

### API Key Security

- The API key is stored in AsyncStorage/SharedPreferences (on-device only)
- The API key is transmitted over HTTPS (Netlify enforces SSL)
- If the key is compromised, the user can revoke it from the SpendWiser Settings page
- Each key maps to exactly one user — it cannot access other users' data

### Privacy

- The SMS filter runs entirely on-device — non-bank messages are never sent anywhere
- The app has no analytics, tracking, or telemetry
- The app is open source — users can verify the code

### Permissions Justification

| Permission | Why |
|---|---|
| `RECEIVE_SMS` | Core functionality — intercept incoming SMS |
| `READ_SMS` | Required alongside RECEIVE_SMS on some Android versions |
| `INTERNET` | Send HTTP POST to webhook |
| `FOREGROUND_SERVICE` | Keep the listener alive in background |
| `RECEIVE_BOOT_COMPLETED` | Auto-start after phone restart |
| `POST_NOTIFICATIONS` | Show foreground service notification |

## 23. Edge Cases & Error Handling

### Edge Case 1: Multi-Part SMS

Long bank messages may arrive as multiple SMS parts. The `BroadcastReceiver` handles this:

```kotlin
val messages = Telephony.Sms.Intents.getMessagesFromIntent(intent)
val fullMessage = messages.joinToString("") { it.messageBody }
```

Always join all parts before filtering or forwarding.

### Edge Case 2: API Key Not Set

If the user hasn't entered an API key yet but SMS is received:
- Log it in the activity log as "⚠️ Skipped (no API key)"
- Do NOT queue it — the user hasn't set up the app yet

### Edge Case 3: Webhook Returns 401 (Invalid Key)

The API key was revoked or regenerated from the web dashboard:
- Show a notification: "⚠️ API Key invalid. Please update your key in the app."
- Stop forwarding until the key is updated
- Do NOT retry — retrying with an invalid key is pointless

### Edge Case 4: Duplicate SMS

Some phones broadcast the same SMS twice:
- Keep a cache of the last 10 SMS hashes
- If the hash matches, skip the duplicate

```typescript
function hashSms(text: string): string {
  let hash = 0;
  for (let i = 0; i < text.length; i++) {
    const char = text.charCodeAt(i);
    hash = ((hash << 5) - hash) + char;
    hash |= 0;
  }
  return hash.toString();
}
```

### Edge Case 5: Very Fast Sequential SMS

If 3 bank alerts arrive within 2 seconds (e.g., EMI deductions):
- Process them sequentially, not in parallel
- Use a simple queue/mutex to avoid race conditions

### Edge Case 6: Phone Has No Internet at SMS Time

This is handled by the offline queue (Feature 6). The SMS is stored locally and retried when connectivity returns.

### Edge Case 7: App Killed by Battery Optimization

Some OEMs (Xiaomi, Samsung, Huawei) aggressively kill background services. Mitigations:
- Use `START_STICKY` for the service
- Request battery optimization exemption
- Guide users to manufacturer-specific settings (e.g., Xiaomi's "AutoStart")

Consider adding a link in the app: [Don't Kill My App](https://dontkillmyapp.com/) — a community resource for OEM-specific battery settings.
